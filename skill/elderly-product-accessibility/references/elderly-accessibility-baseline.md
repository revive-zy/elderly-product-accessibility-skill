# 适老化产品设计基线

版本: 2026-06-15

用途: 用于中老年用户产品的新建、旧产品改造、PRD 编写、UI 规范、开发计划、验收检查和可访问性审查。

## 1. 设计目标

适老化不是把字体放大，而是系统性降低五类负担:

- 视觉负担: 看不清、找不到、颜色难辨、信息太密。
- 认知负担: 不知道当前在哪、不知道下一步、不理解术语。
- 操作负担: 按不准、步骤多、手势复杂、输入困难。
- 信任负担: 怕被骗、怕扣钱、怕泄露、怕点错。
- 恢复负担: 出错后不知道原因，找不到撤销、客服或家人协助。

产品目标是让用户用当前能力完成关键任务，而不是训练用户适应复杂系统。

## 2. 用户分层

按能力分层，不按年龄粗分。

| 用户类型 | 典型能力 | 设计重点 |
| --- | --- | --- |
| 轻度适老用户 | 熟悉微信、短视频、支付，但讨厌复杂规则 | 信息清晰、流程短、权限透明 |
| 中度适老用户 | 会用常见 App，但容易卡在搜索、注册、验证码、表单、支付 | 大入口、强引导、少输入、错误可恢复 |
| 重度适老用户 | 视力、手部、记忆或数字熟练度明显下降 | 大字、大触控、语音、少步骤、人工/家人协助 |
| 协助者用户 | 子女、社区人员、客服、家属 | 授权协助、远程指导、账号找回、配置迁移 |

每个需求必须说明目标层级。不能用同一套复杂度覆盖所有老人。

## 3. 产品原则

### 3.1 任务优先

首页和核心导航按用户任务组织，不按业务频道组织。

优先使用:

- 继续学习
- 查余额
- 预约挂号
- 联系客服
- 查看订单
- 查询结果
- 领取补贴

避免使用:

- 服务大厅
- 权益中心
- 精选频道
- 我的资产
- 成长体系
- 任务广场
- 活动专区

### 3.2 一屏一个主目标

每个核心页面只保留一个主动作。其他动作降级为次级按钮、文字链接、更多菜单或后续步骤。

页面必须回答四个问题:

- 我现在在哪里?
- 这里最重要的信息是什么?
- 我下一步应该点哪里?
- 如果点错或不懂，怎么退出或求助?

### 3.3 默认路径短，进阶路径隐藏

新手和低能力用户走默认路径即可完成任务。进阶能力用户可以通过搜索、筛选、设置、更多功能扩展，但这些入口不能干扰默认路径。

### 3.4 强反馈和可恢复

点击、提交、授权、支付、删除、修改、退出都要有明确反馈。高风险动作必须提供确认、修改、取消、撤销、凭证或客服入口。

### 3.5 不考试化、不惩罚化

学习、健康、福利、服务类产品应避免排名、连续打卡惩罚、失败扣分、羞辱性提示。更适合小步完成、正反馈、可选继续。

## 4. 页面结构

### 4.1 首页

适老化首页应是低心力入口。

推荐结构:

- 顶部: 当前状态、问候、搜索或语音入口。不要放复杂轮播。
- 主区: 3 到 5 个最高频任务入口。
- 主动作: 一个明确按钮，例如"继续学习"、"马上办理"、"查看结果"。
- 辅助区: 客服、家人协助、语音阅读、长辈版设置。
- 底部导航: 3 到 4 个稳定入口，最多不超过 5 个。

禁止把核心入口淹没在广告、运营位、活动卡、信息流、横滑分类中。

### 4.2 列表页

列表项只展示决策需要的信息:

- 名称或主题。
- 当前状态。
- 关键日期、金额、次数或地点。
- 一个下一步动作。

避免每条信息堆叠多个标签、复杂角标、倒计时、营销文案和不可解释的排序。

### 4.3 详情页

按决策顺序展示:

1. 当前结果或状态。
2. 用户下一步能做什么。
3. 为什么是这个结果。
4. 规则、材料、说明。
5. 客服、家人协助、常见问题。

医疗、政务、金融、保险、补贴等场景必须先给结论，再给细则。

### 4.4 表单页

表单要分段、少字段、即时纠错。

要求:

- 一页只处理一类信息。
- 只收集必要字段。
- 每个字段说明为什么需要。
- 手机号、身份证、银行卡、日期、地址自动格式化。
- 验证码提供语音或其他替代方式。
- 错误提示说清原因和改法。
- 长表单支持保存草稿和返回修改。

### 4.5 结果页

结果页必须明确:

- 成功还是失败。
- 已完成什么。
- 后续是否需要等待。
- 凭证在哪里。
- 下一步可以做什么。
- 出错如何处理。

失败页不允许只写"操作失败"、"系统错误"、"参数异常"。

### 4.6 空状态和异常状态

空状态要给原因和下一步，例如:

- "目前没有学习记录，点下面按钮开始今天的 3 个词。"
- "还没有订单。你可以回到首页查看可办理服务。"
- "网络连接不稳定，请检查网络后重试，也可以联系客服。"

## 5. 交互逻辑

### 5.1 操作尺寸

最低要求:

- 适老版界面主要组件可点击区域不小于 60 x 60 dp/pt。
- 单独适老版 App 首页主要组件不小于 48 x 48 dp/pt。
- 其他主要组件不小于 44 x 44 dp/pt。
- Android 通用触控目标建议至少 48 x 48 dp。
- iOS 控件命中区域建议至少 44 x 44 pt。

核心按钮建议高度 56 到 64 dp/pt，按钮之间保持足够间距，危险按钮和确认按钮不要紧贴。

### 5.2 手势

不要把核心功能放在:

- 长按。
- 左滑/右滑。
- 双指缩放。
- 三指及以上手势。
- 悬停。
- 仅键盘快捷键。
- 仅图标按钮。

如果使用手势，必须有可见替代入口。

### 5.3 时间压力

除非任务本身必须限时，否则不要让界面在用户完成操作前自动变化。验证码、付款倒计时、活动倒计时要清楚说明，并提供重发、延期或替代方案。

### 5.4 弹窗

弹窗只用于与当前任务强相关的信息。关闭按钮必须明显，命中区域不小于 44 x 44 dp/pt。不要随机出现广告弹窗、诱导下载弹窗、临时促销弹窗。

### 5.5 状态反馈

所有关键动作都要有反馈:

- 点击中: 按钮状态变化。
- 加载中: 说明正在处理什么。
- 成功: 明确完成结果。
- 失败: 原因、解决办法、重试入口。
- 高风险: 二次确认和撤销。

### 5.6 返回和撤销

返回行为要稳定。高风险动作要支持取消、修改或撤销。法律承诺、财务交易、重要数据提交要优先设计错误预防。

## 6. 信息展示

### 6.1 字体

最低要求:

- 适老版主要文字信息不小于 18 dp/pt。
- 正文建议 18 到 22 sp/pt。
- 核心数字、标题、按钮建议 22 到 28 sp/pt。
- 主要功能和界面文字支持随系统字体调整，或 App 内提供字体大小设置。
- 最大字号下不截断、不重叠、不遮挡主操作。

### 6.2 行距和段落

- 段落内行距至少 1.3 倍。
- 段落间距至少比行距大 1.3 倍。
- 长文案拆段，避免单段超过 3 到 4 行。

### 6.3 对比度

- 普通文本、文本图像、图标等元素对比度至少 4.5:1。
- 大于 18 dp/pt 的大字号文本至少 3:1。
- 禁止只用颜色表达状态，必须同时使用文字、图标、语音或结构变化。

### 6.4 文案

文案要生活化、结果导向、少术语。

替换示例:

| 避免 | 推荐 |
| --- | --- |
| 权益膨胀 | 可领取的好处变多了 |
| 任务激励 | 完成后可获得 |
| 授权链路 | 需要你同意后才能继续 |
| 二次触达 | 之后再提醒你 |
| 提交失败 | 没有提交成功，请检查手机号后重试 |
| 参数错误 | 信息格式不对，请重新填写 |

### 6.5 图标和图片

- 图标必须配文字。
- 装饰图不要干扰信息。
- 使用真实、生活化、不过度年轻化的图片。
- 纯装饰图片应从读屏中隐藏。

## 7. 核心场景规范

### 7.1 注册登录

- 优先手机号一键登录、微信登录或本机号码验证。
- 验证码提供语音或替代方式。
- 登录失败说明原因。
- 找回账号入口明显。
- 不要在非必要场景强制注册。

### 7.2 支付和交易

- 金额、收款方、周期、退款方式、取消方式必须在确认前展示。
- 附加服务不得默认勾选。
- 不得使用诱导付款按钮。
- 提供订单凭证、客服和家人确认入口。

### 7.3 学习和内容产品

- 首页优先"打开就学"，不要让分类导航取代主流程。
- 每日默认小切片，例如 3 个新内容。
- 完成后给正反馈，继续学习作为可选项。
- 不做排名、考试压力、失败惩罚。
- 内容生活化、成人化，不幼稚。

### 7.4 医疗、政务、金融、保险

- 先展示当前结果和下一步。
- 关键材料、日期、金额、风险单独突出。
- 所有提交和授权都有确认页。
- 失败后给明确处理路径。
- 提供人工客服或线下协助说明。

### 7.5 客服和家人协助

- 客服入口在首页、失败页、支付页、授权页、找回账号页可见。
- 机器人不能形成死循环，必须能转人工。
- 家人协助要有明确授权边界，避免家属无感接管用户账号。

## 8. 安全和信任

必须执行:

- 适老版界面和单独适老版 App 不出现广告插件、随机广告或临时广告弹窗。
- 不设置诱导下载、诱导付款等诱导式按键。
- 个人信息处理遵循最小必要原则。
- 权限申请只在需要时出现，并说明用途。
- 付费订阅展示周期、金额和取消方式。
- 高风险动作提供确认、凭证、撤销或客服。

## 9. 旧产品改造方法

### 9.1 P0 改造

先改关键路径，不先做全量视觉翻新。

P0 包括:

- 首页核心入口重排。
- 注册登录、验证码、找回账号。
- 支付、授权、提交、删除等高风险动作。
- 表单和结果页。
- 客服和家人协助入口。

### 9.2 P1 改造

- 列表、详情、搜索、筛选、消息。
- 字号、对比度、间距、按钮尺寸统一。
- 错误和空状态补齐。
- TalkBack/VoiceOver 可读性。

### 9.3 P2 改造

- 设计系统 token。
- 组件库适老化 variant。
- 自动化可访问性检查。
- 埋点分析和老年用户可用性测试。

## 10. 研发落地

### 10.1 Android

- 支持系统字体缩放，避免固定高度文本容器。
- Compose 自定义可点击控件设置足够 `sizeIn` 或最小点击区域。
- 非文本控件设置语义和 `contentDescription`。
- 纯装饰图不暴露给辅助服务。
- 使用 TalkBack 测试核心路径。
- 表单使用明确 label、错误文本和焦点管理。

### 10.2 iOS

- 支持 Dynamic Type 和 Accessibility Size。
- 设置 VoiceOver label、hint、trait。
- 控件命中区域至少 44 x 44 pt，核心操作更大。
- 支持 Reduce Motion 和系统对比度偏好。
- 避免大字号下文本裁剪和按钮挤压。

### 10.3 Web

- 使用语义化 HTML、landmark、heading 层级。
- 支持键盘导航和可见焦点。
- 表单 label 与错误提示可被读屏读取。
- 避免键盘陷阱。
- 符合 WCAG 2.2 AA 的关键条款: 可感知、可操作、可理解、稳健。

## 11. 验收检查表

每个页面至少检查:

- 这个页面的唯一主任务是否明确?
- 用户 3 秒内能否知道下一步点哪里?
- 首页核心入口是否不超过 5 个?
- 主按钮是否足够大，且不与危险按钮紧贴?
- 字号在 1.5x 和 2x 下是否不截断、不遮挡、不重叠?
- 文本对比度是否达到普通文本 4.5:1、大字 3:1?
- 是否没有只用颜色表达状态?
- 是否没有核心功能依赖复杂手势?
- 是否没有非必要倒计时、自动跳转、随机弹窗?
- 错误提示是否说明原因和解决办法?
- 是否支持返回、取消、修改、撤销或客服?
- TalkBack/VoiceOver 是否能读出控件目的和状态?
- 是否有人工客服或家人协助入口?
- 是否没有广告插件、诱导下载、诱导付款?
- 权限和个人信息收集是否最小必要?

## 12. Codex 使用模板

新产品 PRD:

```text
使用 $elderly-product-accessibility，帮我为这个面向中老年用户的产品输出 PRD。
请覆盖用户分层、核心任务、首页结构、核心流程、信息展示、交互反馈、安全信任、研发验收标准。
```

旧产品改造:

```text
使用 $elderly-product-accessibility，审查这个现有 App/页面是否适老化。
请按 P0/P1/P2 输出改造优先级，并给出页面结构、交互、信息展示、可访问性和安全信任问题。
```

UI 规范:

```text
使用 $elderly-product-accessibility，为这个产品制定适老化 UI_SPEC。
请包含字体、行距、触控区域、颜色对比、按钮层级、表单、弹窗、错误态、空状态和辅助技术验收。
```

研发计划:

```text
使用 $elderly-product-accessibility，基于现有 PRD 生成移动端开发计划。
请把适老化要求转换成 Android/iOS/Web 可实现的组件、状态、测试和验收项。
```

## 13. 标准依据

本基线参考:

- 工业和信息化部《移动互联网应用(APP)适老化通用设计规范》相关条目: 字号、行距、对比度、颜色用途、验证码、触控区域、手势、充足操作时间、浮窗、辅助技术、广告和诱导限制、最小必要个人信息处理。链接: https://www.hunan.gov.cn/zqt/zcsd/202104/t20210416_16470434.html
- WCAG 2.2: 可感知、可操作、可理解、稳健，以及对比度、目标大小、错误预防、焦点和辅助技术兼容等准则。链接: https://www.w3.org/TR/WCAG22/
- Android Accessibility: 对比度、至少 48 x 48 dp 触控目标、语义描述和 TalkBack 测试。链接: https://developer.android.com/guide/topics/ui/accessibility/apps
- Apple Human Interface Guidelines: 命中区域至少 44 x 44 pt、文本可读性和触控控件建议。链接: https://developer.apple.com/design/tips/
